Detailed Description

Post-order traversal analysis for basic blocks.

Before use this analysis you have to run doBBsAnalysis().

For this example below analysis returns basic blocks in this order: left, right and bb. As you can see in post-order traversal.

bb:
br i1 1, label %left, label %right
left:
ret i32 0
right:
ret i32 0

Basic blocks can be in strongly connected component. Example below shows this situation. In this situation is not possible to do a correct post-order. It is because basic blocks sccbb and sccbb1 creates strongly connected component. You can use here two types of traversal:

  1. When you use for getting next block only method getNextBB(), than you get basic blocks in this order: end, sccbb1, sccbb, bb. All basic blocks only once.

  2. When you are sure by isNextInSCC() that the next basic block is in strongly connected component than you can use getNextBBInSCC(). This method causes iterating through strongly connected component until you use method stopIteratingSCC(). Then use getNextBB() which returns the next basic block which is out from iterated strongly connected component.

    For example order:

    1. getNextBB() - returns end.
    2. isNextInSCC() => returns true then getNextBBInSCC() - returns sccb1.
    3. 2 times getNextBBInSCC() - returns sccbb and sccbb1.
    4. stopIteratingSCC() and getNextBB() - returns bb.
      bb:
      br label %sccbb
      sccbb:
      br label %sccbb1
      sccbb1:
      br i1 1, label %bb, label %end
      end:
      ret i32 0

llvm::BasicBlock * bin2llvmir::BBTraversalAnalysis::getNextBBInSCC ( )

Returns next basic block that is part of some SCC.

You have to use this method only if you know that next returned basic block will be part of some SCC. For this purpose to check use method isNextInSCC(). Than you can use this method more than once. To stop iterating through basic blocks in some SCC use stopIteratingSCC().
